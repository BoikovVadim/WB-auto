# Стратегия управления рекламными кластерами WB

> Живой документ. Источник истины по логике автоматического управления ставками и составом
> рекламных кластеров. Статус: дизайн согласован, реализация по этапам за флагом через preview.

## Цель

**Максимум заказов при эффективности.** Держать работающим максимум кластеров и позиций, пока
это окупается (CPO ≤ Макс СРО), и не переплачивать там, где заказы дала бы органика или меньшая ставка.

## Базовые понятия

| Понятие | Определение |
|---|---|
| Заказы | `max(ordРК, ordJAM)` — **везде единообразно** (в CPO, CR, показ-заказ) |
| CPO (СРО) | расход / заказы |
| Макс СРО | цена × %выкупа × ДРР × 2 — **единый порог** (×2 целевого уже внутри, повторно не умножаем) |
| CR (показ→заказ) | `max(ordРК, ordJAM) / max(показы, 100)` — пол 100 показов гасит шум малых кластеров |
| **bid_cap** | `Макс СРО × 1000 × CR` — фиксированная формула, макс. ставка, при которой CPO = Макс |
| Накопит. счётчики | по ключу `(advert, nm, cluster, ценовой_бакет)`: spend, orders_rk, orders_jam, views; децей с полураспадом 90 дней |
| Позиция | зонд `u-search` (выдача с рекламой); цель — **топ-4** (коридор 4–5) |

`max(ordРК, ordJAM)` берётся везде осознанно (risk-on): CR с органическими заказами выходит
оптимистичной, `bid_cap` щедрее — принято ради потенциала доходности, т.к. чистых данных WB не отдаёт.

## Лестница состояний кластера (первое сработавшее сверху)

1. Ручное `blacklisted` / `protected` — абсолют, движок не трогает.
2. Новый от ВБ (`pending`) → авто-OFF, ждёт решения человека.
3. Бесзаказный → LEARNING / отключение.
4. Заказный → ставочный движок.
5. Смена цены → переоценка (переключение ценового бакета).
6. ДРР-регулятор → выключение по ранжированию.

## 1. Старт
Все кластеры включаются на **минимальной ставке**.

## 2. Бесзаказные (заказов нет)
- накопленный расход без заказа **< Макс СРО** → **LEARNING** (держим на минималке, копим данные);
- **≥ Макс СРО** → **отключаем**.

У нормального кластера (истинный CPO ≈ целевой) на расходе = Макс СРО ожидается 2 заказа, значит
P(0 заказов) = e⁻² = **13.53 %** — приемлемая доля «невезучих» отключений. Разведка идёт сама на
минимальной ставке: карточка имеет общую силу ранжирования, поэтому бесзаказный кластер всё равно
получает показы за счёт позиций по другим кластерам.

## 3. Заказные — ставочный движок
**Квалификация:** `accrued_orders (rk+jam) > 0` в актуальном ценовом бакете. Навсегда, без свежести —
редкий малопоказный кластер (заказ раз в 2–3 месяца) остаётся в игре.

**Потолок `bid_cap = Макс СРО × 1000 × CR`** — самонастройка окупаемости:
- показы копятся без заказов → CR↓ → `bid_cap` → минимум (неокупаемый оседает сам);
- мало показов + редкий заказ → CR держится → потолок высокий;
- пришёл заказ → CR↑ → потолок восстановлен.

**Позиционный регулятор** (каждые 10 мин, по кругу, без приоритета, без сглаживания — решение по
каждому замеру):
- `P > 5` (выпали) → **поднять** ставку ∝ (P − 4), но не выше `bid_cap`;
- `P ≤ 4` → **probe-down**: снижаем пробным шагом, ищем минимально достаточную ставку (экономия —
  органика растёт, конкуренты меняются);
- шаг **асимметричный**: вниз осторожно, вверх агрессивно ∝ дистанции;
- на `bid_cap` и топ-4 не достали → **стоим**, ждём halo (реклама → заказы → растёт органика →
  позиция подтянется, потом снизим ставку);
- на минимуме и `P ≤ 5` → стоим.

**Отключение:** `bid_cap < мин ставки` ⟺ даже на минимуме CPO > Макс (CR слишком низкая) →
отключаем (ранний предиктивный рез по конверсии). Активный зонд-обход — только кластеры с
`bid_cap > мин` (где есть что регулировать); осевшие на минимум выходят из круга, но остаются
квалифицированными.

## 4. ДРР-регулятор (дневной) — жёсткий ограничитель расхода, резка по ранжированию

План ДРР (7.20 %, настраиваемый) — **жёсткий ограничитель дневного расхода** (режем даже рентабельных,
если надо уложиться). Ставки заказных автономны — ДРР-регулятор ими не управляет, он только **выключает
кластеры по ранжированию**. `cutoff`/плавающий порог не используется (опускание ставок сползало бы ниже
топ-4 у всех; прямое выключение целиком сохраняет топ-4 у выживших).

**Перетрата** (ДРР выше плана) — выключаем целиком снизу вверх, пока Σ срезанного дневного расхода ≈
рублёвый избыток `(вчерашний расход − план × выручка)`:
1. убыточные заказные (`bid_cap < мин`) — обычно уже сняты базовым правилом;
2. бесзаказные по убыванию расхода (большой → мелкий, даже мелочь);
3. рентабельные заказные по возрастанию `bid_cap` (худшая конверсия первой).

Выжившие держат свои ставки и топ-4.

**Недотрата** (ДРР ниже плана) — включаем обратным порядком: рентабельные с бóльшим `bid_cap` → … →
бесзаказные мелкие → крупные (мелкие первыми — на тот же бюджет включится больше кластеров = шире
разведка). Низкий ДРР — это хорошо, не доедаем расход как самоцель.

Стабилизация: мёртвая зона ±0.50 %, неполный шаг 0.65. Приоритет ДРР-выключения над ставочным движком —
выключение держится флагом до следующего дневного пересчёта.

## 5. Смена цены
Изменение ≥ ~5 % → новый ценовой бакет → счётчик пуст. **Наследуем CR из соседнего бакета** как
стартовую оценку (конверсия слабо зависит от ±5 % цены), чтобы не падать на минималку с нуля.
Снижение цены воскрешает кластеры в LEARNING, Макс СРО пересчитывается. Старые бакеты лежат —
вернулись на цену, получаем готовый результат без пересбора.

## 6. Возврат отключённых
Накопительные счётчики → CPO меняется плавно. После отключения заказы продолжают капать (РК — окно
атрибуции; органика — через JAM), а показы заморожены → `max(ordРК, ordJAM)` ↑ → CPO ↓ и CR ↑ →
`bid_cap` поднимается → кластер **возвращается сам**. Ожидание бесконечное (halo: реклама сеет
органику).

## Данные и инфраструктура
- Позиция → зонд `u-search` (топ-100, ~1 с/кластер).
- CR → `accrued_views` (источник `/adv/v0/normquery/stats` или fullstats) + `accrued_orders`.
- Макс СРО → `ProductCpoService`.
- Ставка → `wb_cluster_bids`, `desiredBid`, очередь `bid-write`, `/adv/v0/normquery/bids` (готово).

## Масштаб и производительность
Заказных кластеров ~1 000–1 900. Круг на одном зонде ~17–31 мин (растянутый круг принят). Движок
мерит длительность круга, число write и долю 429 от `bid-write`; превышение целевого времени или рост
429 → явный сигнал «добавить параллельный зонд/IP или батчинг» (не деградирует молча).

## Порядок внедрения
За флагом через preview по этапам: `accrued_views` → `bid_cap` → позиционный регулятор →
ДРР-ранжирование → телеметрия. Каждый этап обкатывается на данных (движок считает, WB не трогает) →
перевод на live. Флаг = рубильник отката.

Открытые параметры реализации: коэффициент `k` шага ставки, значение минимальной ставки, лимит
`bid-write` (закроется телеметрией).

## Аудит логики

### Раунд 1 (метрики) — закрыт
Деление на ноль/шум CR (пол 100 показов), разведка на минималке (отклонено — общая сила карточки),
возврат выключенного (работает через дозревание заказов), приоритет ДРР над ставочным, CR не делит
рекл/орг (max-заказы, risk-on), инерция памяти (децей 90 дней), смена цены (наследуем CR бакета),
CR↔позиция↔bid_cap (консервативно, общая CR). Подробности — в истории решений.

### Раунд 2 (системные/процессные) — закрыт
- **Бюджет кампании WB** — на автопополнении WB, движок им не управляет (не наша забота).
- **ДРР по всей выручке** (не отделяем рекламную) — halo взаимный (реклама бустит органику и
  наоборот), а чистых данных WB не отдаёт; разделение бессмысленно и невозможно.
- **Отказоустойчивость зонда:** при 429/сбое — пауза и **продолжение с того же кластера** (круг
  не сбрасывается, лишь растягивается), цикл замкнут; при отсутствии позиции ставки замораживаются
  как есть, ДРР-регулятор (не зависит от зонда) продолжает работать.
- **Атрибуция заказов** — доверяем данным WB (другого источника нет).
- **Окна (накопленный CPO vs вчерашний ДРР)** — реактивно по вчерашнему дню: плохой день может
  временно срезать рентабельный кластер, система вернёт его при недотрате. Так и задумано.
- **Ручной оверрайд ставки** — нет, всё автомату (ручные только `blacklisted/protected` на вкл/выкл).
- **Clamp ставки:** `ставка = clamp(желаемая, мин, min(bid_cap, max_WB))` — заложить (краевой:
  при хорошей конверсии товар близок к топу и до max WB обычно не доходит).
- **Единый план ДРР и цель топ-4** для всех — старт; маржа/юнит-экономика считается отдельно и
  остаётся управленческим решением человека (вне системы). Задача движка — выжать максимум заказов
  в рамках плана ДРР.
- **Состав кластеров:** кластер пропал из выдачи WB → замораживаем его счётчики и убираем из
  активного обхода (новый от WB → `pending` → авто-OFF, уже есть).
